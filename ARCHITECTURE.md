# Hairo — Инженерные правила бэкенда

> Фиксирует инженерные правила проекта: организация кода, модули, работа с БД, DTO/контракты/REST API, тесты.
> Адаптировано из проверенного ruleset другого проекта. Бизнес — `PROJECT.md`, продукт — `PRODUCT.md`, стек/вопросы — `TECH.md`, план — `ROADMAP.md`.

---

## 1. Архитектурные решения (зафиксировано)

- **backend:** NestJS
- **frontend:** Nuxt.js
- **база данных:** PostgreSQL
- **ORM:** TypeORM
- **паттерн доступа к данным:** Repository Pattern
- **Telegram Bot:** grammY (long polling)
- **шаблонизатор текстов/ответов бота:** Handlebars (шаблоны в `src/{module}/templates/`)
- **платежи:** Stripe
- **очередь задач:** генерация асинхронная → очередь (например BullMQ + Redis)
- **хранение изображений:** S3-совместимое объектное хранилище (регион ЕС), отдача через CDN

Изменение `TypeORM`, `grammY` или базовой слоистой структуры требует отдельного архитектурного решения.

**Документация через context7:** для NestJS, grammY/Telegram SDK, Stripe SDK, TypeORM и любых внешних библиотек сначала получать актуальную документацию через MCP `context7`, а не полагаться на память модели. Это основной источник истины по API, конфигурации, decorators, plugins, CLI и паттернам интеграции.

## 2. Слои backend

Каждый доменный модуль строится по трём слоям:

- **controllers** — маршрутный слой, guards, чтение входных данных, вызов сервиса, трансформация ответа в DTO
- **services** — бизнес-логика и прикладные сценарии
- **repositories** — инфраструктурный доступ к БД через `Repository<Entity>`

> **Исключение — модуль `telegram`:** не имеет HTTP-контроллеров. Вместо них grammY-хендлеры в `services/handlers.service.ts`. Слои service и repository — в обычном виде.

Жёсткие правила:
- контроллеры не содержат бизнес-логику
- сервисы не работают с HTTP-деталями
- сервисы не пишут SQL напрямую
- репозитории не содержат бизнес-логику
- репозитории работают через API `Repository<Entity>`
- `queryBuilder` — только в крайних случаях, когда стандартного API недостаточно
- у всех членов классов явные модификаторы видимости (`public`/`private`/`protected`); enforced ESLint-правилом `@typescript-eslint/explicit-member-accessibility` (конструкторы — без `public`)

## 3. Структура модулей

Доменный принцип:

```text
src/{module}/
  {module}.module.ts
  constants/
    index.ts
  dao/
    {entity}.entity.ts
    migrations/
      {timestamp}-{migration-name}.ts
  controllers/
    {module}.controller.ts
  dto/
    {dto}.ts
  repositories/
    {module}.repository.ts
  services/
    {module}.service.ts
  templates/            # если модуль формирует тексты через Handlebars
```

Правила:
- имена файлов и классов отражают домен
- несколько контроллеров/сервисов/репозиториев — внутри тех же директорий
- миграции хранятся по единому правилу во всём проекте

### Предполагаемая доменная декомпозиция Hairo (черновик, уточняется)

- `users` / `auth` — аккаунты, аутентификация (Telegram initData + email)
- `telegram` — grammY-хендлеры, авторизация Mini App, бот/уведомления
- `photos` — загрузка, хранение, улучшение исходного селфи
- `face-analysis` — vision-анализ селфи (форма лица, длина/текстура/густота волос, цвет) → префилл визарда и фильтра каталога
- `catalog` — типы причёсок, таксономия, теги, референсы
- `generation` — пайплайн генерации (очередь, статусы, результаты)
- `billing` — Stripe, кредиты/баланс, история списаний

**Инфраструктурные модули** (не домены, без бизнес-логики и HTTP-контроллеров с доменными эндпоинтами, инкапсулируют конкретный внешний SDK): `storage` (S3), `image-model` (генерация изображений, Google AI), `llm-model` (structured vision-анализ, Google AI). Доменные модули обращаются к ним через DI и не работают с внешними SDK напрямую (см. `TECH.md`).

## 4. TypeORM и база данных

Сущности — в `src/{module}/dao`. Репозитории инжектят `Repository<Entity>` через `@InjectRepository`.

Допустимые базовые операции: `find`, `findOne`, `findAndCount`, `insert`, `update`, `save`.

Правила:
- типизированные сущности
- связи описывать явно
- принадлежность данных владельцу учитывать в сервисах и запросах
- изменение схемы — только через миграции
- миграции писать через **TypeORM API** (`queryRunner.createTable(new Table(...))`, `createIndex(new TableIndex(...))`, `dropTable`/`dropIndex`), не сырым SQL; в `down()` откатывать всё созданное

Настройки БД и `DataSource` — в `database/` (корень проекта).

## 5. DTO и трансформация ответов

DTO — HTTP-граница приложения.

Правила:
- входные DTO валидируются через `class-validator`
- выходные DTO не содержат бизнес-логику
- контроллер трансформирует результат сервиса в DTO через общую утилиту `transformToDto` (`_common/utils`)

Контроллер: принять request DTO → вызвать сервис → вернуть response DTO. Не собирать ответ вручную, если можно общей трансформацией.

> jsonb/freeform-объекты в DTO отдавать через `@Transform(({ obj }) => obj.field, { toClassOnly: true })` — иначе `excludeExtraneousValues` рекурсивно вычищает вложенный объект.

## 5a. Контракты (единый источник типов backend ↔ frontend)

Слой контрактов — в `api/src/_contracts`. Единый источник типов внешнего API для backend и frontend; собирается в отдельный npm-пакет `@hurgadan/hairo-contracts` и публикуется в GitHub Packages.

Структура:
```text
api/src/_contracts/
  package.json        # @hurgadan/hairo-contracts (publishable)
  tsconfig.json       # NodeNext, declaration, outDir dist
  api-base.ts
  localized-text.type.ts
  index.ts            # реэкспорт всех модулей
  {module}/
    index.ts
    {entity}.type.ts
    body-{action}.type.ts | create-{entity}.type.ts
    {module}-api.ts   # abstract class с baseUrl и сигнатурами эндпоинтов
```

Правила:
- контракты — только `interface`/типы и abstract `*-api.ts` классы (без runtime-логики);
- **backend DTO реализуют контракты через `implements`** (компилятор держит DTO в синхроне с API);
- backend импортирует контракты по относительному пути к файлам типов: `../../_contracts/{module}/{x}.type` (не из корня папки — у неё свой `package.json`, dist локально не собран);
- frontend потребляет опубликованный пакет `@hurgadan/hairo-contracts`;
- модульные `index.ts` реэкспортируют модуль, корневой `index.ts` — все модули;
- публикация: `.github/workflows/publish-contracts.yml`, версия — через `.github/scripts/bump-version.sh` (feat → minor, иначе patch).

## 6. Общие константы и утилиты

Магические значения не допускаются.

- константы модуля — `src/{module}/constants/index.ts`
- общие типы — `_common/types`
- общие утилиты — `_common/utils`
- общая инфраструктура — `_common/app`

Тексты и локализация:
- пользовательские тексты не хардкодятся бесконтрольно в бизнес-логике; **i18n RU/ES/DE с первого дня**
- коды статусов/типов/системных значений — стабильные константы, независимые от языка отображения

Шаблоны уведомлений/текстов бота — через Handlebars, в файловой системе, не в коде.

## 7. Доступ к данным

- каждый пользователь видит только свои данные
- все доменные сущности привязаны к владельцу
- проверки принадлежности не живут только в контроллерах — фильтрация по владельцу в сервисах и при необходимости в репозиториях
- доступ к чужим фото, аккаунтам, балансу, генерациям запрещён

## 8. Чувствительные данные (GDPR)

Фото лиц = персональные данные, биометрия особо чувствительна в ЕС.

- явное согласие на обработку, прозрачная политика, право на удаление, минимизация
- сроки хранения исходных и сгенерированных фото
- хранилище предпочтительно в ЕС-регионе
- проверить условия модели генерации (Google AI) по использованию загруженных данных для обучения (opt-out)

## 9. Тестирование

Обязательно: **unit** и **e2e**.

**Unit:** тестируют бизнес-логику; внешние зависимости и репозитории мокаются.

**e2e:**
> **КРИТИЧНО:** e2e **НИКОГДА** не запускать напрямую на локальной машине (`npm run test:e2e`). Тесты чистят таблицы и уничтожат локальную dev-базу. Запуск **только** через Docker-окружение с изолированным Postgres из `docker-compose`.

- изолированная среда, отдельный PostgreSQL в `docker-compose`, база на `tmpfs`
- перед запуском применяются миграции; перед каждым тестом таблицы очищаются
- каждый тест создаёт только нужные данные; данные через фабрики/общие утилиты
- тестовые утилиты: очистка таблиц, создание тестового приложения и HTTP server, получение `Repository<Entity>`

Тесты и миграции запускаются в CI при push и merge в `main`.
