## Хранение фото — S3-совместимое хранилище (зафиксировано)

Единый протокол (S3 API) на всех окружениях, переключение — только через env, без изменения кода. Абстракция в бэкенде: `StorageModule` (global) + `StorageService` на `@aws-sdk/client-s3` (`api/src/storage`).

- **Локальная разработка:** **MinIO** в `docker-compose` (`db:up`), S3 API на `:9000`, консоль на `:9001`. Бакет создаётся автоматически при старте (`onModuleInit`, только при `STORAGE_FORCE_PATH_STYLE=true`).
- **Продакшн:** **Cloudflare R2** (бакет с `jurisdiction: eu` — data residency под GDPR/биометрию; бесплатный egress; free tier покрывает старт; встроенный CDN). `region=auto`, `FORCE_PATH_STYLE=false`, endpoint `https://<account>.eu.r2.cloudflarestorage.com`, `STORAGE_PUBLIC_URL` = CDN-домен бакета.
- **Альтернатива для прода:** Hetzner Object Storage (Германия/ЕС, €4.99/мес) — если хочется один поставщик/DPA с VPS; минус — платный egress при росте.
- **Отдача:** публичный CDN-URL через `STORAGE_PUBLIC_URL`; если не задан — presigned URL (приватные объекты). Клиентская загрузка — через presigned PUT (browser → хранилище напрямую).
- **Env:** `STORAGE_ENDPOINT`, `STORAGE_REGION`, `STORAGE_ACCESS_KEY_ID`, `STORAGE_SECRET_ACCESS_KEY`, `STORAGE_BUCKET`, `STORAGE_FORCE_PATH_STYLE`, `STORAGE_PUBLIC_URL` (см. `.env.dist`).

## Интеграция с AI-моделями (зафиксировано)

Два независимых инфраструктурных модуля (`api/src/{module}`), у каждого свой клиент SDK — общий инстанс не шарим:

- **`image-model`** — генерация/рестайлинг изображений. Провайдер — Google AI (`@google/genai`), модель — Nano Banana / Gemini 2.5 Flash Image. Абстракция `BaseModel` + `createModel`/`itIsMe` — на будущее переключение между версиями модели через env (`IMAGE_MODEL`).
- **`llm-model`** — структурированный vision-анализ фото (форма лица, длина/текстура/густота волос и т.п. → JSON по схеме). MVP — только Gemini (`generateContent` + `responseSchema`), тот же паттерн абстракции, что и в `image-model`. LLM как класс задач — не только Gemini: большинство провайдеров (кроме Anthropic) совместимы с OpenAI-протоколом, что в будущем упростит переключение провайдера через env. На этапе MVP лишние интеграции не тащим — только Gemini.

Доменные модули (например `face-analysis`) обращаются к `llm-model`/`image-model` через DI, не работают с SDK напрямую — так же как `photos` не работает с `@aws-sdk/client-s3` напрямую, а идёт через `StorageService`.

## Асинхронные шаги (анализ фото, генерация) — паттерн (зафиксировано)

Для MVP — без очереди/Redis. Контроллер создаёт запись задания со статусом `pending`, кидает fire-and-forget async-вызов (обязательно с `.catch()` — иначе unhandled rejection может уронить процесс, и статус останется зависшим), отвечает `202 Accepted`. Фронт поллит `GET /{module}/:id` до `completed`/`failed`.

Первое применение — `face-analysis` (см. `ROADMAP.md` Фаза 3). Статус-модель задания спроектирована на переиспользование в `generation` (Фаза 2/3), когда там появится реальная очередь (BullMQ/Redis) — заменится только механизм старта задания, API для фронта (create + poll) не изменится.
