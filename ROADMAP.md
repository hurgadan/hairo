> Фазовый план (вехи). Детальные таски внутри фазы дробим, когда доходим до неё — раньше они «плывут».
> Контекст: `PROJECT.md` (бизнес), `PRODUCT.md` (продукт), `ARCHITECTURE.md` (правила бэкенда), `TECH.md` (вопросы).

**Принцип:** строим тонкими вертикальными срезами. Сначала сквозной пайплайн на плейсхолдер-промптах, потом наращиваем. Качество промптов — отдельный непрерывный трек, не блокер.

## Фаза 1 — Технический фундамент

Цель: каркас по `ARCHITECTURE.md`, на котором всё дальше растёт.

- [x] NestJS-скелет (запускаемый каркас, HTTP + pino + ValidationPipe + QueryFailedFilter + Swagger, `/health`), `_common` ← билд зелёный
- [x] PostgreSQL + TypeORM, `DataSource`, первая миграция (`catalog`) — `migration:run` отрабатывает
- [x] DTO + `transformToDto`, `class-validator` (применено в `catalog`, `auth`)
- [x] `auth`/`users`: JWT + email/пароль (bcrypt) + Telegram initData (HMAC) + `JwtAuthGuard` + `/auth/me` — проверено вживую
- [x] `docker-compose` — dev Postgres (`db:up`, проект `hairo`) + изолированный тестовый Postgres на tmpfs (`docker:test`, проект `hairo-test`)
- [x] Тестовый контур e2e (`catalog` + `auth`, 8 тестов, на тест-утилитах) + CI (GitHub Actions: lint · build · e2e на postgres-сервисе)
- [x] Nuxt.js-скелет (`web/`): Nuxt 4 + Tailwind v4 + токены `DESIGN.md`, страница каталога из API — проверено сквозь (SSR рендерит каталог из БД)
  - follow-up: фронт должен потреблять `@hurgadan/hairo-contracts` — нужен токен/секрет с `read:packages` (сейчас 403), пока локальная копия типа в `web/app/types`

**Выход:** «пустое, но правильное» приложение с зелёным CI.
